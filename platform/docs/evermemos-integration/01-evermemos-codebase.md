# EverMemOS Codebase Deep-Dive Report

> Generated by evermemos-codebase-expert agent (50+ source files read)

## 1. Architecture Summary

EverMemOS is an enterprise-grade long-term memory system for conversational AI agents. It claims 93% accuracy on the LoCoMo benchmark. The system ingests raw conversation messages, detects conversation boundaries, extracts multiple memory types, and stores them in a triple-write architecture (MongoDB + Milvus + Elasticsearch).

### Tech Stack
- **Language**: Python 3.12 (uv package manager)
- **Web Framework**: FastAPI + Uvicorn
- **ODM**: Beanie (async MongoDB ODM built on Motor)
- **Vector DB**: Milvus 2.5.2 (HNSW index, cosine metric)
- **Full-text Search**: Elasticsearch 8.11
- **Cache**: Redis 7.2
- **LLM**: OpenAI-compatible API (OpenRouter by default, aiohttp client)
- **Embedding**: vLLM primary, DeepInfra fallback
- **Reranking**: Qwen3-Reranker-4B (vLLM primary, DeepInfra fallback)
- **Tokenizer**: tiktoken o200k_base
- **Chinese NLP**: jieba (for BM25 tokenization)

### Layered Architecture (outside-in)

```
API Layer (FastAPI controllers)
  -> Service Layer (DI-managed services)
    -> Agentic Layer (retrieval strategies, reranking, vectorization)
      -> Business Logic Layer (memorize flow, DB operations, sync)
        -> Memory Layer (extraction, clustering, profiling, LLM, prompts)
          -> Infrastructure Layer (MongoDB docs, Milvus collections, ES indices)
            -> Core Framework (custom DI container, middleware, events, tenants)
```

### Custom DI Framework
EverMemOS has a Spring-like DI container (`core/di/container.py`) with:
- Decorators: `@component`, `@service`, `@repository`, `@controller`, `@mock_impl`, `@factory`, `@conditional`, `@depends_on`
- Scopes: SINGLETON, PROTOTYPE, FACTORY
- Auto-wiring via constructor parameter type annotations (inspects `__init__` signature)
- Mock mode toggle for testing
- Bean ordering strategy with priority: mock > direct match > primary > factory
- Circular dependency detection via resolving stack
- Inheritance-aware lookup (ABC interface -> implementation classes)

Startup: `run.py` -> `application_startup.setup_all()` -> loads addon entry points -> `setup_dependency_injection()` scans and registers all beans.

## 2. Data Flow

### Write Path (Message Ingestion)

```
HTTP POST /api/v1/memories
  -> MemoryController.memorize_single_message()
    -> convert_simple_message_to_memorize_request() [request_converter.py]
    -> MemoryRequestLogService.save_request_logs() [audit trail]
    -> agentic_layer.MemoryManager.memorize()
      -> biz_layer.mem_memorize.memorize()
        -> preprocess_conv_request() [validates, normalizes]
        -> memory_layer.MemoryManager.extract_memcell()
          -> ConvMemCellExtractor.extract_memcell()
            -> Force-split if >8192 tokens or >50 messages
            -> LLM boundary detection (5 retries, JSON parse)
            -> Returns: MemCell + StatusResult(should_wait)
        -> If boundary detected (not should_wait):
          -> _save_memcell_to_database() [MongoDB, publishes MemCellCreatedEvent]
          -> process_memory_extraction():
            -> PARALLEL extraction:
              - Episode extraction (LLM -> title/content + embedding)
              - Foresight extraction (up to 10 predictions + embeddings)  [assistant scene only]
              - EventLog extraction (atomic facts + embeddings)           [assistant scene only]
            -> Update MemCell with extracted memory references
            -> _trigger_clustering() [semantic clustering of MemCells]
              -> If cluster has enough memcells: _trigger_profile_extraction()
            -> save_memory_docs() [MongoDB + sync to ES + Milvus]
        -> If should_wait: message stays pending for next boundary
```

### Read Path (Memory Retrieval)

```
HTTP GET /api/v1/memories/search
  -> MemoryController.search_memories()
    -> convert_dict_to_retrieve_mem_request()
    -> agentic_layer.MemoryManager.retrieve_mem()
      -> Dispatches by RetrieveMethod:
        - KEYWORD:  ES BM25 search (jieba tokenization for Chinese)
        - VECTOR:   Milvus cosine similarity search
        - HYBRID:   keyword + vector + rerank (Qwen3-Reranker-4B)
        - RRF:      Reciprocal Rank Fusion (k=60) of keyword + vector
        - AGENTIC:  LLM-guided multi-round retrieval with sufficiency check
      -> group_by_groupid_strategy() [group results by group_id]
      -> Return RetrieveMemResponse with scores
```

### Fetch Path (Key-Value Lookup)

```
HTTP GET /api/v1/memories
  -> MemoryController.fetch_memories()
    -> agentic_layer.MemoryManager.fetch_mem()
      -> FetchMemService.find_memories()
        -> Queries MongoDB by user_id/group_id/memory_type/time_range
```

## 3. Memory Types

| Type | Description | Storage | Embedding |
|------|-------------|---------|-----------|
| **MemCell** | Atomic unit; a bounded conversation segment | MongoDB only | No |
| **EpisodeMemory** | Narrative summary of a MemCell (title + content) | MongoDB + ES + Milvus | Yes |
| **Foresight** | Predictive memory (up to 10 per MemCell) with time range | MongoDB + ES + Milvus | Yes |
| **EventLog** | Atomic facts extracted from conversation | MongoDB + ES + Milvus | Yes (per fact) |
| **Profile** | User characteristics (GROUP_CHAT scenario) | MongoDB (UserProfile) | No |
| **ProfileLife** | Explicit info + implicit traits (ASSISTANT scenario) | MongoDB (UserProfile) | No |
| **GroupProfile** | Group-level topics, roles, member analysis | MongoDB | No |

### Scene-Based Behavior
- **ASSISTANT scene**: Extracts Episode + Foresight + EventLog. Clones group episodes to each participant user. Triggers ProfileLife extraction (explicit_info + implicit_traits).
- **GROUP_CHAT scene**: Extracts group + personal Episodes only. Triggers standard Profile extraction (work-oriented).

## 5. Storage Details

### MongoDB (Source of Truth)
- **Collections**: `memcells`, `episodic_memories`, `foresight_records`, `event_log_records`, `user_profiles`, `group_user_profile_memories`, `cluster_states`, `conversation_metas`, `memory_request_logs`
- **ODM**: Beanie documents with `DocumentBase` / `DocumentBaseWithSoftDelete`
- **Soft Delete**: `deleted_at` field; composite indexes include `deleted_at` for query optimization
- **Indexes**: Composite on (user_id, timestamp), (group_id, timestamp), (group_id, user_id, timestamp), keywords, linked_entities

### Milvus (Vector Search)
- **Index type**: HNSW (M=16, efConstruction=200)
- **Metric**: COSINE
- **Dimension**: From `VECTORIZE_DIMENSIONS` constant (configurable)
- **Collections**: `episodic_memory`, `foresight`, `event_log`
- **Fields**: id, vector, user_id, group_id, participants (ARRAY), event_type, timestamp, episode, search_content, metadata, parent_type, parent_id

### Elasticsearch (BM25 Full-text)
- **Indices**: `episodic-memory`, `foresight`, `event-log`
- **Analyzers**: `whitespace_lowercase_trim_stop_analyzer` for title/episode/summary; `standard` for search_content
- **Chinese tokenization**: App-level jieba segmentation (not ES plugin)
- **Tenant-aware**: Uses `TenantAwareAliasDoc` for multi-tenant index routing

### Redis (Cache)
- Used via `RedisProvider` (DI-managed)
- Role: Request deduplication, caching

## 6. LLM Integration Points

### LLM Calls (via `memory_layer/llm/openai_provider.py`)
All LLM calls go through `OpenAIProvider` using aiohttp to call OpenAI-compatible APIs:
- **Boundary Detection**: Determines if accumulated messages form a complete conversation segment
- **Episode Extraction**: Converts MemCell into narrative summary (title + content)
- **Foresight Extraction**: Generates predictive memories from conversation
- **EventLog Extraction**: Extracts atomic facts
- **Profile Extraction**: Extracts user characteristics (standard or life profile)
- **GroupProfile Extraction**: Analyzes group dynamics (topics, roles)
- **Agentic Retrieval**: Multi-round LLM-guided search with sufficiency checking
- **Multi-query Generation**: Generates search query variations for agentic retrieval

### Key LLM Config
- Provider: `LLM_PROVIDER` (default: "openai")
- Timeout: 600 seconds
- Retry: 5 attempts with random 5-20s sleep on 429 errors
- Optional provider ordering via `LLM_OPENROUTER_PROVIDER`

### Embedding (via `agentic_layer/vectorize_service.py`)
- Primary: vLLM endpoint
- Fallback: DeepInfra
- Failure tracking: switches to fallback after 3 consecutive primary failures
- Batch embedding support

### Reranking (via `agentic_layer/rerank_service.py`)
- Model: Qwen/Qwen3-Reranker-4B
- Primary: vLLM; Fallback: DeepInfra
- Same resilience pattern as vectorize service

## 10. Surprises and Notable Findings

1. **Empty MCP adapter**: `infra_layer/adapters/input/mcp/` is completely empty - planned extension point, natural kagent integration surface.

2. **Dual-write complexity**: Every memory type writes to MongoDB AND syncs to both ES and Milvus. No saga/compensation mechanism - if sync fails, search indices may be inconsistent.

3. **App-level Chinese tokenization**: jieba tokenization happens at the application layer, not via ES plugin.

4. **1400+ line monolithic files**: `mem_memorize.py` (1413 lines) and `mem_db_operations.py` (1352 lines) are the core write path.

5. **Scene-specific pipelines**: ASSISTANT extracts 3 types in parallel (Episode + Foresight + EventLog); GROUP_CHAT extracts Episodes only. ASSISTANT also clones group episodes to individual users.

6. **Clustering triggers profiling**: Profile extraction is NOT on the direct write path - triggered when a semantic cluster accumulates enough MemCells.

7. **Resilient embedding/reranking**: Primary/fallback pattern with max 3 failures before switching. Production-hardened.

8. **`timeout_to_background` decorator**: Memorize endpoint can push long-running extraction to background processing if HTTP request times out.

9. **`MAGIC_ALL` constant**: Sentinel value for "match all" - skip filtering on that field during queries/deletions.

10. **No authentication**: Memory controller uses `default_auth="none"`. Auth framework exists but not wired.

11. **Soft delete**: MemCells support soft delete via `deleted_at` timestamps. All queries must filter on `deleted_at=None`.
