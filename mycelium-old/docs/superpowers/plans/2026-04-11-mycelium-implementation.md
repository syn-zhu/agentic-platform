# Mycelium Implementation Plan

**Goal:** Implement the Mycelium vertical slice — one OAuth CredentialProvider (GitHub), one Tool (list_repos), full 3LO flow from agent tool discovery through credential injection using MCP [URL mode elicitation](https://modelcontextprotocol.io/specification/draft/client/elicitation#url-mode-elicitation-for-oauth-flows).

**Architecture:** Three Go components sharing CRD types:
- **Controller** — watches Mycelium CRDs, generates AGW resources (MCP backend, tool-access policies, JWT validation) + Knative Services + credential broker configuration
- **Engine** (sidecar) — MCP server behind AgentGateway (via UDS), ext_proc for inbound sessions, pod claimer. Calls credential broker over RPC for token management.
- **Credential Broker** (shared service) — OAuth callbacks, session binding, token vault (MongoDB + CSFLE), on-demand refresh

**Spec:** `docs/superpowers/specs/2026-04-11-mycelium-design.md`
**Auth design:** `docs/Magenta-auth.md`

## Milestone 1: CRDs + Controller

Build the controller and resource generators. Pure K8s API work, testable with envtest.

### Scope
- CRD types: Project (formerly TenantConfig), CredentialProvider (formerly OAuthResource), Tool, Agent
- Resource generators: AGW MCP backend (UDS), HTTPRoute, JWT policy, source context policy, tool-access policy, Knative Service
- Reconcilers: Project, CredentialProvider, Tool, Agent — with phase-based reconciliation, deferred status patching, sub-conditions
- Validating webhooks: dependency checks on create/update/delete

### Completion criteria
- [ ] `make generate` produces CRDs and deepcopy
- [ ] All reconcilers create correct AGW + Knative resources (envtest)
- [ ] Webhooks reject invalid operations (unit tests)
- [ ] Status conditions reflect sub-resource health (ProjectValid, CredentialsValid, ServiceReady, etc.)

### Design considerations
- Tool-access policy uses `backend.mcp.authorization` with `mcp.tool.name` CEL — generated by inverting the agent→tool mapping
- AGW backend uses `unixPath` for UDS transport to the engine
- CredentialProvider reconciler should push provider config to the credential broker (TBD: via CRD status, direct RPC, or ConfigMap)

### Status: **In progress** (CRDs, generators, reconcilers, and webhooks are implemented)

## Milestone 2: Credential Broker

Build the shared credential broker service. Owns all OAuth lifecycle operations.

### Scope
- Token vault: MongoDB CSFLE store with per-tenant KMS isolation
- RPC interface: `GetToken(tenant, agent, user, resource, scopes)` → token or auth-required response
- OAuth callback handler: `/oauth2/callback/{provider}` — receives auth codes from external providers
- Session binding: redirect to developer-managed verifier, receive confirmation via `/oauth2/confirm`
- Token exchange: auth code → tokens, store in vault
- On-demand refresh: transparent to callers, inline on expired tokens
- Completion notification: notify engine when token is stored (engine forwards `notifications/elicitation/complete` to agent)

### Completion criteria
- [ ] Token store/retrieve/refresh cycle works with real MongoDB + CSFLE (integration test)
- [ ] Per-tenant key isolation: tenant A cannot read tenant B's tokens
- [ ] OAuth callback → session binding → token exchange flow works end-to-end (integration test with mock OAuth provider)
- [ ] Concurrent `GetToken` calls for same tuple return same pending `elicitationId` (OQ-1)
- [ ] Refresh failure deletes vault entry and returns auth-required

### Design considerations
- **MongoDB architecture**: per-tenant database with isolated CSFLE key vault collection. Per-tenant MongoDB credentials scoped to `readWrite` on their database only.
- **Provider registration**: controller pushes provider config (client ID, endpoints, secret ref) to the broker. Mechanism TBD — could be direct RPC, a CRD the broker watches, or a ConfigMap.
- **Callback URL generation**: broker needs a stable public URL. Could be a dedicated Ingress or a path on the AGW external listener proxied to the broker.

### Race conditions / edge cases
- Concurrent `GetToken` calls triggering duplicate OAuth flows for the same tuple
- Token refresh race: two concurrent requests both find an expired token and try to refresh
- Callback received before verifier confirmation (hold in pending state)
- Verifier confirmation timeout (pending codes expire after TTL)
- MongoDB unavailable during token lookup (return retryable error, not auth-required)

## Milestone 3: Engine — MCP Server + Credential Integration

Build the engine as an MCP server that serves tools and calls the credential broker for tokens.

### Scope
- MCP server (mcp-go SDK) serving tools from Tool CRDs, behind AGW via UDS
- Identity resolution: read AGW-injected headers (`X-Source-Pod-IP`, `X-Source-Service-Account`), resolve via informer cache
- Tool invocation: call credential broker → inject token into execution context → forward to Tool Executor's Knative Service
- Elicitation: return `URLElicitationRequiredError` when broker indicates auth required; send `notifications/elicitation/complete` when broker notifies completion

### Completion criteria
- [ ] Agent can `tools/list` through AGW and see only permitted tools
- [ ] Agent can `tools/call` and receive result with token transparently injected
- [ ] Agent receives `URLElicitationRequiredError` when no token exists
- [ ] Agent receives `notifications/elicitation/complete` after OAuth flow completes
- [ ] Retry after completion notification succeeds with token injected

### Design considerations
- Engine communicates with AGW over UDS (`/shared/agent/engine.sock`)
- Engine communicates with credential broker over internal RPC (gRPC or HTTP)
- Tool execution context body format (similar to Arcade's format: tool name, inputs, context with authorization token)
- AGW tool-access policy filters `tools/list` responses automatically — engine serves all tools

### Race conditions / edge cases
- Elicitation completion arrives while agent has already timed out and retried
- Multiple agents in the same session invoke different tools requiring different providers simultaneously
- Tool executor timeout / failure after token injection (token was valid but tool failed)

## Milestone 4: Engine — ext_proc + Pod Claiming

Build the ext_proc handler for inbound session establishment and the pod claimer.

### Scope
- Informer cache: watch Sandbox/SandboxClaim CRDs, build pod IP → (agent, session, user) map
- Pod claiming: SSA SandboxClaim with deterministic name `claim-{agent}-{hash(session-id)}`
- ext_proc gRPC service: handle inbound requests on AGW external listener, resolve/create sandbox claims
- Routing: always use Service FQDN for routing, never pod IP directly

### Completion criteria
- [ ] Inbound request claims a sandbox pod (or reuses existing) via SSA
- [ ] Subsequent requests from same session route to same sandbox
- [ ] Informer cache correctly reflects SandboxClaim state
- [ ] Released pod is never routed to (Service FQDN stops resolving)

### Design considerations
- Cache is an existence check only — routing uses deterministic Service FQDN
- SSA with deterministic name is idempotent under concurrent requests
- ext_proc must be low-latency (informer cache for fast path, SSA only on cache miss)

### Race conditions / edge cases
- Two concurrent requests for a new session both trigger SSA — idempotent, both wait for Ready
- Stale cache shows claim exists but it was released — route to Service FQDN, DNS fails, clean error
- Cache miss for existing claim — SSA is a no-op, claim becomes visible in cache on next sync
- Pod warmup latency on first request (no warm pool hit)
- SandboxClaim stuck in non-Ready state (timeout + error response)

## Milestone 5: End-to-End Integration

Wire up all components and prove the full vertical slice.

### Scope
- Deploy controller, credential broker, engine sidecar, AGW, and Knative in a test cluster
- One working CredentialProvider (GitHub) + one Tool (list_repos) + one Agent
- Full 3LO flow: tool discovery → tool invocation → elicitation → OAuth → session binding → retry → success

### Completion criteria
- [ ] E2E test proving:
  1. Apply Project, CredentialProvider (GitHub), Tool (list-repos), Agent
  2. Controller generates: Knative Service, AGW MCP backend, tool-access policy
  3. Agent calls `tools/list` → sees `list_repos` (filtered by AGW)
  4. Agent invokes `list_repos` → engine returns `URLElicitationRequiredError`
  5. Simulate OAuth completion (mock provider callback + verifier confirm)
  6. Engine receives completion notification, sends `notifications/elicitation/complete` to agent
  7. Agent retries `list_repos` → token injected → Tool Executor calls GitHub API → result returned
- [ ] Controller, credential broker, and engine all start cleanly and recover from restarts

### Open questions to resolve during implementation
- **OQ-1:** Single active elicitation flow per `(tenant, agent, user, resource, scopes)` tuple? (Leaning yes)
- **OQ-2:** Credential broker deployment — dedicated Ingress for callbacks, or proxied through AGW external listener?
- **OQ-3:** Provider config push mechanism — controller → broker via RPC, CRD watch, or ConfigMap?
- **OQ-4:** MongoDB deployment topology — shared Atlas cluster with per-tenant CSFLE isolation, or per-cell instance?
